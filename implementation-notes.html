Design Decisions

1. Single endpoint with format=count instead of a sub-action

The diagnostic floated two shapes for the badge count: ?action=tasks&scope=badge or ?action=tasks&format=count. Picked the latter. Same endpoint serves list and count, just with different output shape. Matches the existing pattern of ?action=attachments&id=X (one action, modifier param). Avoids action proliferation.

2. TasksView placement — sub-view of Prospects, not a new top-level tab

Added as a third option in the existing Table / Charts sub-view toggle inside the Prospects tab. The "My Tasks" badge in ProspectFilters switches the sub-view to Tasks. Rationale: keeps Brett in the Prospects workflow rather than navigating away; one less top-level tab to add; the Charts toggle pattern already exists so the visual affordance is familiar. A future global Header bell can open the same component if needed.

3. Activity-log entries on task lifecycle

Create, mark-done, dismiss, reopen, delete each emit one activity-log entry. Edits to description, due_date, or assignee on an open task do not log — quick reschedules would otherwise flood the activity feed. Status transitions are the audit-worthy events.

Lifecycle eventActivity log text
CreateTask created: <description>
Mark done✓ Task completed: <description>
Dismiss✗ Task dismissed: <description>
Reopen↺ Task reopened: <description>
Delete⌫ Task deleted: <description>

4. Task lifecycle does NOT update suggested_next_step

The existing add-activity auto-overwrite of prospect_companies.suggested_next_step is preserved verbatim (api/prospects.js:2766-2771). Task-lifecycle log entries insert into prospect_activity_log but do not touch suggested_next_step. Otherwise creating a task would clobber whatever manual next-step note Brett has on the prospect, which violates the locked spec point 8.

5. Assignee dropdown source

Fetches the existing active users (4) via GET /api/auth?action=list-users, which is admin-only. Solution: lift the list of users onto the auth context at login (it's already there as useAuth().user for the current user, but not as the team list). Decision: TasksView/TaskInlineEditor fetch the team list once on mount via a new lightweight ungated endpoint? No — to avoid adding endpoints, I'll hard-code the 4 active users (Kyle, Duane, Steve, Brett) as a fallback constant in taskUtils.js, and additionally try to fetch from a new endpoint ?action=team-members that returns name+color for active users (one extra branch in api/auth.js, no new function). The dropdown also includes "Unassigned" (maps to NULL).

Update: Used the new api/auth.js?action=team-members branch. Auth-gated but returns just { name, color } for active users. No admin role required.

6. Dismissed tasks visibility

Hidden by default in the per-prospect Tasks section (rendered as a single "Show N completed/dismissed" expandable footer). Visible in TasksView when the user picks the "Completed" or "Dismissed" status chip.

7. Tasks column in ProspectTable replaces the Due column

The "Due" column (read from follow_up_date) is removed from the table. The new "Tasks" column shows count of open tasks with a small urgency dot for the earliest due date. Sort: by earliest open task due_date ASC, NULLS LAST. Companies with zero open tasks sink to the bottom.

The follow_up_date column stays in the schema. Reads remain functional for the urgency function (which still treats it as Tier 1) — but the UI no longer surfaces it for editing in ProspectDetail. New work flows entirely through tasks.

8. Action items badge re-labeled "My Tasks"

The red pill at the top-right of ProspectFilters is preserved visually but its count semantics change: (assignee = currentUser OR assignee IS NULL) AND status = 'open'. Click switches the sub-view to Tasks with the default filter. The preset button "Action Items" in the same component is renamed "My Tasks" and is now identical to clicking the badge.

Deviations

  • Stale preset stays. Spec says staleness moves out of the badge but the "Stale" preset already exists separately in ProspectFilters and was untouched. This is consistent with the spec point 4 ("staleness indicators move out of the badge and into a separate 'stale prospects' filter that already exists").
  • follow_up_date editor is removed from ProspectDetail. Spec says "UI stops reading it (deprecated in display)". I removed the date-picker block (lines 638-683). The column itself remains in the schema. The urgency function still reads it client-side for the legacy "Stale" preset compatibility (Tier 1 of getProspectUrgency), so the existing 3 follow-up rows still render their urgency until the migration script populates equivalent tasks.

Tradeoffs

  • Single SQL count vs separate assigned/unassigned queries. Picked a single SQL with WHERE (assignee = $1 OR assignee IS NULL) AND status = 'open'. Two queries would have surfaced "Brett has 5 of his own + 3 unassigned", but the badge has room for one number, not two, and the spec is unambiguous about combining.
  • Hard delete vs soft delete. Spec allows both. Implemented hard delete in the API behind an explicit confirm prompt. Primary affordances in the UI are Done and Dismiss; Delete is a less prominent action under a per-row overflow.
  • Per-prospect task count fetch. Considered two options: (a) one query returning all open tasks grouped by prospect_id (single round-trip), (b) include a task_count subquery in the existing prospect list endpoint. Picked (a) — keeps the prospect list endpoint untouched and lets the Tasks column be a self-contained client concern.

Open Questions

  • Task ordering on group header rows. The Tasks column on a parent-group header currently shows only the parent's own open task count, not aggregated across subsidiaries (unlike Sites/Acquisitions which aggregate). Confirm this is the right default — aggregation is a one-line change in the renderer if Brett prefers it.
  • Hard-delete affordance. The trash-can button is exposed on every TaskRow behind a confirm prompt. Should this be admin-only? Current spec says "supported but prefer dismissed", which I read as "available to everyone, just not primary." Will defer to Brett's preference once he sees the UI.
  • Migration apply timing. Recommendation in the wrap-up: Kyle reviews the dry-run output first, then runs `--apply`. There are only 3 candidate rows so this is low-risk.
  • Activity log filtering. Task lifecycle entries now mingle with manual activity entries in the same `prospect_activity_log`. If the feed becomes noisy after team adoption, we could add a filter chip ("hide task lifecycle") — but that's a future enhancement, not part of this build.
