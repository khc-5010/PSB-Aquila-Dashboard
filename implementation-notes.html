This file replaces the prior "Threads 2+3" notes (those remain in git history on branch claude/adoring-cori-rynr9) — it is the per-task communication channel for this feature.

Design Decisions

1. Server-side assembly (new action=export-json), not client-side stitching

The payload must follow corporate links and include each linked record's attachments / activity / tasks. Doing that client-side would mean many round-trips (the detail view doesn't load linked entities or their sub-data). A single server endpoint assembles the whole graph in a handful of queries and returns one clean object. Lives as a new action branch inside api/prospects.js (no new file — Vercel 12-function limit; "one file per feature"), using the same sql client already in the file.

2. SELECT p.* for company rows (schema-agnostic)

tasks/lessons.md warns that scripts/create-prospect-table.sql is stale vs. the live schema (~52 columns live). Using SELECT * / SELECT p.* captures whatever columns actually exist, so the export never drifts from the live schema and I don't hardcode a column list. Mirrors the existing single-prospect read at api/prospects.js:2356.

3. Link walk: typed links + former_names rows, 1-hop, cycle-guarded

Per user decision. Relationships are denormalized strings, so matching is case-insensitive (LOWER(TRIM(...))), consistent with the import upsert and the table grouping. Three sources, deduped by row id (typed label wins over former_name):

SourceMatchrelationshiplink_basis
Typed childrenparent_company = primary.company AND parent_relationship_kind IN (subsidiary, absorbed_into)subsidiary / absorbed_intoparent_company
Typed parentcompany = primary.parent_company (only if primary has a typed kind)parentparent_company
Former namescompany = ANY(primary.former_names)former_nameformer_names

This catches Sybridge → X-Cell whether X-Cell is wired as a typed child (parent_company='SyBridge Technologies') or only listed in Sybridge's former_names[]. financial_sponsor (PE) siblings are deliberately excluded — lessons.md confirms parent_company/sponsor is overloaded with ~200 PE strings; the typed-kind filter is what keeps the walk tight. Cycle guard: a visited Set seeded with the primary id; 1-hop only (no recursion into linked entities' own links); cap 25 linked entities (linked_entities_truncated: true if exceeded).

4. Contacts: contacts: [] placeholder, real names ride in free text

Per user decision and Phase-1 finding: there is no structured contacts table for prospects. The workable contacts (e.g. Jaeger, Przybylski under X-Cell) live only in company.notes, company.psb_connection_notes, and research_brief attachments. Because the export includes the full linked X-Cell record plus its attachments, those names travel with the payload — the empty contacts array is a forward-compatible placeholder, documented as not-yet-modeled.

5. activity_log[].type is derived, best-effort

The table has no type column. I derive it from entry-text prefixes (matching the lifecycle symbols documented in the prior notes): Task…/✓/✗/↺/⌫ Task → task; ⚑ / ✓ Review → flag; else note. Divergence: the target shape listed status_change, but status changes are not individually logged (only reflected in updated_at), so that value never appears. Raw entry, author, and timestamp are always included so nothing is lost.

Deviations

  • No live-DB verification in this container. lessons.md confirms there is no DATABASE_URL here. In-env verification is therefore: node --check (server syntax), npm run build (client), and scripts/verify-export.js --mock (a Sybridge fixture run that proves the link-walk/dedup/shape and prints the assembled payload). Live-data confirmation is handed to the user as a read-only Neon SQL query + a deployed-endpoint test (URL mode of the same script).
  • Replaced (not appended) implementation-notes.html. The prior file documented the unrelated Threads 2+3 work; this is a fresh per-task channel. Prior content is preserved in git history.

Tradeoffs

  • Inlined assembly logic vs. importable helper. A shared, unit-testable module isn't possible: Vercel serverless can't import from src/, and adding a file under api/ would create a phantom 13th function. So the assembly lives inline in the endpoint, and scripts/verify-export.js carries a faithful reference implementation of the same algorithm for fixture testing (cross-referenced in comments to guard drift).
  • Batched = ANY(${allIds}) sub-entity reads vs. per-record loops. Chose 3 batched queries (attachments / activity / tasks) over 3 × N round-trips. Proven idiom in this file (api/prospects.js:1413). Keeps a deep export well within the 10s Vercel timeout.

Open Questions

  • None blocking. If duplicate-name rows exist (lessons.md notes ~35), a former_names string could resolve to more than one row; both are included (deduped by id). Acceptable for an export, but flagging in case you want a tighter match later.

Option B — read-only Neon SQL (paste into the Neon SQL editor; confirms the live link graph the walk depends on):

WITH primary AS (
  SELECT id, company, parent_company, parent_relationship_kind, former_names
  FROM prospect_companies WHERE company ILIKE '%sybridge%'
)
SELECT 'children' AS link, c.id, c.company, c.parent_relationship_kind
FROM prospect_companies c, primary p
WHERE LOWER(TRIM(c.parent_company)) = LOWER(TRIM(p.company))
  AND c.parent_relationship_kind IN ('subsidiary','absorbed_into')
UNION ALL
SELECT 'former_name' AS link, c.id, c.company, NULL
FROM prospect_companies c, primary p
WHERE LOWER(TRIM(c.company)) = ANY(SELECT LOWER(TRIM(x)) FROM unnest(p.former_names) x)
  AND c.id <> p.id;
