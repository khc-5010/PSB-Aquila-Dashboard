Design Decisions

1. former_names type: TEXT[] (not JSONB)

Postgres TEXT[] is the native fit for an ordered list of plain strings. It supports = ANY(former_names) and ARRAY_TO_STRING(former_names, ', ') directly, which we'll use for the searchable index in Phase 3. JSONB would only be worth the complexity if we needed per-entry metadata (source, date_absorbed) — and the diagnostic didn't surface a need for that. If richer per-entry data becomes necessary later, migrating TEXT[] to JSONB of objects is a clean ALTER + transform.

Convention check: the codebase uses JSONB only for users.digest_preferences (a settings blob, multiple fields per row); no existing TEXT[] usage was found, but that's because no field has previously needed multi-valued semantics. former_names is the natural first use.

2. parent_relationship_kind values

The column is plain TEXT with no CHECK constraint — UI dropdowns govern valid values. This keeps future additions (e.g. 'spinoff_of', 'joint_venture') friction-free. Initial enum surface for Phase 3 UI:

  • 'subsidiary' — operational parent; brand keeps identity (Barnes → Synventive).
  • 'absorbed_into' — parent absorbed the brand; the row's identity is the absorbing entity (SyBridge ← X-Cell).
  • 'sister_company' — same parent as another row, no direct parent-child relationship.
  • NULL — independent / no parent.

Phase 2 heuristic backfill emits only 'subsidiary' and 'absorbed_into'. 'sister_company' requires reading parent_company values across rows — better suited to a separate manual triage pass than a noisy heuristic.

3. financial_sponsor is free text (not a controlled list)

Matches the existing free-text pattern of parent_company (and the prevailing convention across this codebase — only ownership_type is loosely controlled, and even that drifts). Promoting financial sponsors to a controlled list would require a sponsors table + FK + UI work that exceeds Phase 1's scope. Light normalization (TRIM, no casing enforcement) is sufficient for now. A future pass can introduce a sponsors lookup if PE-level analytics demand it.

4. former_names import delimiter: pipe |

also_known_as historically uses mixed delimiters — commas, slashes, semicolons — which are also common inside company names ("Sumitomo Bakelite Co., Ltd.", "F&S Tool", "ABC / DEF"). Pipe is unambiguous and easy to type. BulkImportModal.jsx and scripts/seed-prospects.js both parse pipe-delimited cell values into TEXT[]; a single name (no pipe) becomes a one-element array; empty becomes NULL.

5. parent_company and also_known_as kept untouched in Phase 1

Per the prompt: additive schema only. The existing columns continue to read/write exactly as before. Phase 3 will change how the UI interprets them (e.g., GitMerge driven by parent_relationship_kind, not isChild) without dropping the columns themselves. Deprecation, if pursued, is a later cleanup pass after the migration completes and downstream consumers (CSV exports, external reports) are confirmed safe.

6. ONTOLOGY_FIELDS includes the new columns, but Layer-1 logic is unchanged in Phase 1

Adding the new columns to ONTOLOGY_FIELDS means PATCH edits to them retrigger rebuildOntologyForProspect(). The rebuild itself, in Phase 1, still only emits subsidiary_of from parent_company — so the trigger does nothing meaningful yet. This is deliberate: Phase 3 expands the rebuild to branch on the new columns. Triggering early keeps the timing right (no PATCH dropped on the floor when Phase 3 lands).

Tradeoffs

Backfill heuristics — false-positive vs false-negative bias

The heuristic dryrun is biased toward false negatives over false positives. It only flags candidates that match multi-signal patterns (e.g., a parent_company name match alone isn't enough for "high confidence" PE classification — we also want ownership_type to corroborate). The cost: some real PE sponsors won't get auto-flagged and will need manual classification. The benefit: rows that are auto-suggested are the ones Brett can rubber-stamp; the dryrun output won't be a wall of guesswork to wade through.

Expected hit rates against the 934-row dataset (from the diagnostic): roughly 100–150 PE candidates flag at high+medium confidence (vs ~200 PE-shaped parent_company values in total — the rest need manual classification); ~30–50 absorbed candidates (vs 131 both-populated records — the rest are the brand-list cases or ambiguous); ~30–60 subsidiary candidates (the strong cases from Westfall/Barnes/Nolato); ~5–15 brand-list aka cleanup candidates (Barnes is the canonical case, plus a few others).

Why two routes (dryrun + apply) instead of one

Separating the read (backfill-parent-types-dryrun) from the write (apply-parent-type-classifications) protects against accidental writes during exploration. Kyle hits the dryrun route as many times as he likes (or via the candidates JSON saved at checkpoint); the write route only fires when he's reviewed the JSON and is ready. The write route accepts the (potentially edited) candidates JSON as its POST body, so Brett's edits flow through one path.

Why no auto-apply at checkpoint

This is the whole point of the checkpoint. The 131 both-populated records, the ~200 PE strings, and the Barnes-style aka stuffing all need human judgment — especially in the medium/low confidence bands. Auto-applying would risk corrupting rows that the heuristic can't safely classify. The dryrun + checkpoint is the explicit safety valve.

SQL vs JS heuristic fidelity

scripts/backfill-parent-types-dryrun.sql and the api/prospects.js JS handler implement the same heuristic — but they use different regex flavors (POSIX \m/\M for word boundaries in SQL vs JS \b). I cross-checked: POSIX ~* with \m...\M behaves identically to JS /.../.i with \b for the patterns used here (alphanumeric word boundaries). The CASE logic in SQL precisely mirrors the JS branch tree, including the "added_to_A" flag that emulates JS's continue after Category A so Category B and C don't double-classify the same row. If the heuristic needs to change later, edit both files (mark with a SYNC comment).

Phase 3 Requirements (Locked)

The candidate review revealed structural patterns that the Phase 3 UI must accommodate. Locked decisions from the consolidated payload review:

1. financial_sponsor AND former_names are orthogonal dimensions

The original Option A framing implicitly treated the three relationship facets as alternatives — "this row is either a subsidiary OR absorbed OR PE-sponsored." The candidate review surfaced ~27 rows where a PE-owned company also carries absorbed-brand history (SyBridge / X-Cell is the canonical example: Crestview owns SyBridge, and SyBridge absorbed X-Cell). For these rows the apply payload sets BOTH financial_sponsor (Crestview) AND parent_relationship_kind = 'absorbed_into' + former_names = ["X-Cell Tool & Mold"].

Phase 3 detail page UI contract: the ProspectDetail panel must render financial_sponsor and former_names as two independent fields, both visible simultaneously, not as mutually exclusive states. The header should display "Formerly: …" sourced from former_names (regardless of financial_sponsor), and the PSB/Relationship section should display "Financial Sponsor: …" (regardless of former_names). The "Parent Company" field shows only when parent_relationship_kind is 'subsidiary' (operational parent) — for absorbed_into rows the parent is the absorbing entity and remains meaningful; for financial_sponsor-only rows parent_company is NULL and the field hides.

2. Grouping scope (re-confirmed at apply time)

Rows form parent groups only when parent_relationship_kind IN ('subsidiary', 'absorbed_into'). Rows with financial_sponsor populated and parent_company = NULL do NOT group together — even if many rows share the same sponsor string (e.g., the dozen Crestview portfolio companies). PE-portfolio companies are operationally independent and rendering them as a parent group would mislead.

3. Parenthetical-stripping convention for former_names

When a former name is sourced from also_known_as, strip trailing parenthetical annotations like "(acq 2022)", "(rebranded)", abbreviations like "(MDI)". These are commentary on the relationship, not part of the entity's name. The consolidated payload applied this rule to rows 105 (Exo-s → "Plastiques Moore"), 203 (Cosmetic Specialties → "Canopy Beauty Packaging"), and 647 (Aptyx → "Molded Devices Inc.", "Southwest Mold"). If future imports or manual edits introduce parenthetical aka values, downstream code (or import scripts) should apply the same strip.
