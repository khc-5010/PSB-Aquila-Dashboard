Design decisions

Scoring exemption via outreach_group = 'Infrastructure'

The server scorer (api/prospects.js, mirrored in src/utils/priorityScore.js) exempts a prospect from the converter rubric when outreach_group === 'Infrastructure' or category ∈ {Knowledge Sector, Hot Runner Systems, Catalog/Standards, Strategic Partner} → priority_score = NULL, ai_readiness = 'exempt'. Both rows use outreach_group='Infrastructure' — the prospects-side home of the other equipment/infrastructure players (RJG, DME, Husky, Mold-Masters, Beaumont). This is the only mechanism that keeps them unscored, which the brief required.

Category labels (confirmed with Kyle)

Arburg → Strategic Partner; Blue Moose → Ecosystem. Both are existing category values, so no code changed. (category is free TEXT, not an enum/CHECK — a brand-new value would have been safe too, falling to the 'Other' parent group, but existing values keep filters/legend grouping clean. 'Strategic Partner' is itself an exempt category; 'Ecosystem' maps to the existing 'Ecosystem/Channel' parent group.)

prospect_status = 'Outreach Ready'

The "Add to Pipeline" promote button only appears when status is 'Outreach Ready' or 'Converted'. Since the whole point is for Kyle to promote them, both rows are seeded at 'Outreach Ready' (accurate — intro emails already sent), rather than the default 'Identified'.

No inflated signals

signal_count = 0 for both — they are partner/ecosystem contacts, not converters; the value is captured in category + notes, never an invented signal score.

Deviations from the brief

"Kistler is already in the system tagged PARTNER" — not as a prospect row. A full-repo scan found no prospect_companies row for Kistler; it appears only in docs, CLAUDE.md, and cavity-pressure logic. "Tagged PARTNER" refers to the opportunities side (lead_type='partner'). On the prospects side, the real precedent for an equipment/infrastructure partner is the Infrastructure outreach_group — that's what Arburg mirrors. (Net effect is the same intent the brief asked for.)

I did not perform the live insert myself this session. This sandbox had no DATABASE_URL and the egress proxy blocks raw-TCP Postgres; Neon was also not in the allowlist (egress test returned 403). DB access was configured as an env-secret, which applies only to a new session. So the deliverable is two validated, runnable scripts; the live run happens in a fresh session (or via the Neon SQL Editor). The SQL was instead validated end-to-end against a throwaway local Postgres 16 (see Verification).

Tradeoffs & judgment calls

  • Execution channel. A SQL script (Neon SQL Editor) is the guaranteed, zero-dependency path; the Node/HTTP-driver script is what leverages the DB access being set up and lets me run + verify directly. Both produce identical data and are idempotent; both are committed. The deployed API single-create path was rejected because it cannot set prospect_status or parent_relationship_kind (so it couldn't make the rows promotable).
  • Infrastructure group for a micro vendor. Blue Moose is an ecosystem/relationship contact, not "infrastructure" in the RJG/Husky sense — but Infrastructure is the system's lever for "don't score as a converter." Category 'Ecosystem' carries the honest label; outreach_group='Infrastructure' carries the exemption. (Open to revisiting — see Open Questions.)
  • ownership_type = 'Private' for Arburg (vs. "Family/Founder-Owned"). Both are accurate (Hehl & Keinath family), but "Family/Founder-Owned" + 30+ years would trigger the table's family-succession urgency icon — misleading for a stable German Mittelstand parent. "Private" keeps indicators quiet; family ownership is in notes.
  • key_certifications left null. Arburg is "ISO-certified across quality/environment/energy" but the specific standards are unconfirmed. Per the brief's "leave null rather than guess," the column is null (so the ontology doesn't generate guessed Certification entities); the ISO note lives in notes.
  • US-entity vs. parent metrics. The row is "Arburg, Inc." (US subsidiary): year_founded=1990 (US entity), employees/revenue left null (US-entity figures unknown). Parent scale (~3,000–3,400 staff, ~€750M, founded 1923) is recorded in notes, not conflated into the US row's metric columns.
  • parent_relationship_kind='subsidiary'. Generates a clean subsidiary_of ontology edge to the German parent. With only one child and no parent prospect row, it does not create a parent grouping in the table (per the grouping rules), so no visual surprise.

Ontology Layer-1

Layer-1 derivation is application-level only (it runs inside the API handlers — there is no DB trigger), so a raw SQL/HTTP insert does not auto-generate ontology nodes. This is documented, not a bug. After the rows exist, regenerate via POST /api/prospects?action=rebuild-ontology-layer1 (authenticated), or simply edit any ontology field on each row (a PATCH fires rebuildOntologyForProspect). For these light-data rows the only edges are Arburg → subsidiary_of → ARBURG GmbH + Co KG and the two ownership-structure edges — non-critical; they'll also appear on the next bulk import/rebuild.

The brief's data table says Priority: HIGH, but the formula yields 32 → WATCH.
