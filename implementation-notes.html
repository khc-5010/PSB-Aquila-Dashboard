What was built

  • A POST ?action=assistant arm inserted in the POST block, immediately after rebuild-ontology-layer1. It validates the body, builds the system prompt (+ prospect-context line), runs the Anthropic Messages API tool-use loop via plain fetch, and returns { answer, toolsUsed }.

Contract (matches Prompt 2)

Key decisions

DECISION Auth: no new code — the file-level guard already covers it

Recon found api/prospects.js:861-864 runs requireAuth for every action except GET daily-digest. A new POST action is therefore auth-gated automatically. Adding a second requireAuth call inside the arm would be redundant (and the authUser from the file-level call is block-scoped, so it isn't reachable here anyway — and Prompt 1 needs no DB writes, so no attribution is required).

DECISION Provider: Together.ai (OpenAI-compatible), model deepseek-ai/DeepSeek-V3

Plain fetch to https://api.together.xyz/v1/chat/completions with Authorization: Bearer ${TOGETHER_AI_API} (house style; the Resend call at ~1062 is the fetch template). OpenAI-compatible shape: the system prompt is the first messages entry (not a top-level field); tools are mapped to {type:'function', function:{name, description, parameters}} at call time (the neutral defs keep input_schema, mapped to parameters); tool calls return on choices[0].message.tool_calls with finish_reason === 'tool_calls'; each result goes back as a separate {role:'tool', tool_call_id, content} message; arguments arrive as a JSON string and are JSON.parsed (guarded). No SDK, no streaming (single JSON response). The five SQL executors are provider-agnostic — unchanged by the switch; only callModel + the endpoint/model/auth constants changed.

DECISION Tool-use loop & cap behaviour

  • stop_reason === 'tool_use' → append the assistant turn verbatim (carries tool_use blocks), run each tool read-only, append a single user turn of tool_result blocks, loop.
  • Terminal (end_turn / max_tokens / stop_sequence / refusal) → join text blocks into answer; refusal returns a canned safe line.
  • On the MAX_TURNS cap, one final call with tool_choice: {type:'none'} forces a text wrap-up (tools stay in the payload so the prior tool turns in history remain valid), returning the best answer + a "(reached the reasoning-step limit.)" note.
  • Failed tool → tool_result with is_error:true and a safe {error} (never a raw stack). Each stringified tool_result is hard-capped at 12000 chars as a backstop on top of the per-tool bounds.

Deviations from spec (small, intentional)

  • DEVIATION MAX_TURNS wrap-up uses tool_choice:{type:'none'} instead of omitting tools. The spec said "one final call with tools omitted." Omitting tools while the message history still contains tool_use/tool_result blocks risks an API validation error; keeping tools present and forcing tool_choice:none achieves the same "must produce text" outcome safely. Same observable behaviour.
  • DEVIATION search_prospects free-text matches company + notes only (per the tool description), not the wider client-side search field set. Kept narrow to match what the model is told the tool does.

Tradeoffs

  • TRADEOFF get_prospect returns the full company row (explicit columns), which is verbose, vs. a hand-picked few. Chose completeness because the assistant's job is sizing-up/comparison and the row is bounded; the brief is excerpted (500 chars) and contact notes capped (200 chars) to control tokens.
  • TRADEOFF Single JSON response, no streaming. Simpler, and it matches the serverless "await everything before res.end()" rule. A typing indicator on the client covers the wait (Prompt 2).
  • TRADEOFF ontology-neighborhood is deliberately not a tool — it needs an ontology_entities id, not a prospect id. find_similar_prospects covers prospect-centric comparison directly.
