Each finding below was verified by reading the actual code paths; items that need a runtime or live-DB check to fully confirm are marked **SUSPECTED**. Everything else is **CONFIRMED**.

**Topline:** ~100 findings. 6 critical, ~15 high. The good news: the hard algorithmic core (sort/tiebreakers, parent grouping rules, priority score math, category rules, `parseLocalDate` discipline, export-json link walking, SQL parameterization) verified clean. # A. Bugs & Defects

## A1. Critical

### A1.1 — The entire data API is unauthenticated
Every endpoint except `api/auth.js`'s admin actions and the CRON_SECRET check on `daily-digest` (`api/prospects.js:656-659`) accepts requests with **no token at all**. That includes: list/read/create/PATCH prospects, **bulk import upsert**, attachment/task/activity CRUD, `flag-for-review`, ontology rebuild, `save-state-report`, `seed`, `export-json`, `data-audit`, plus all of `opportunities.js`, `opportunities/[id].js`, `activities.js`, `stage-transitions.js`, `key-dates.js`, `analytics.js`. Anyone who knows or guesses the deployed URL can read every research brief and contact note, rewrite the prospect database, or delete opportunities. The login screen protects the UI only — `App.jsx` even fetches `/api/opportunities` successfully *while the login screen is showing*.
**Fix:** shared `requireAuth(req)` helper (validate Bearer token against `sessions` like `auth.js:25-34` already does) at the top of every data handler; client already has `authFetch`. Exempt only the cron action.

### A1.2 — Kanban drag-and-drop corrupts `stage` with a card UUID
`App.jsx:140` uses `const newStage = over.id`, but every card is itself a droppable target (`useSortable`, inside `SortableContext` at `DroppableColumn.jsx:89`). Dropping a card on/near another card (including reordering within a column) makes `over.id` an **opportunity UUID**, which passes the `stage !== newStage` guard, is optimistically applied, and is **persisted** — because the PATCH handler (`api/opportunities/[id].js:131-148`) never validates against `VALID_STAGES` (only POST does, `api/opportunities.js:55`). The card disappears from every column (filtered by real stage keys) until someone fixes the row in SQL. `stage_transitions` also logs `to_stage: <uuid>`.
**Fix:** resolve the container — `over.data.current?.sortable?.containerId ?? over.id` — and bail unless it's a valid stage key; add `VALID_STAGES` validation to the PATCH handler.

### A1.3 — Layer 1 ontology rebuild silently destroys Layer 2 data
`rebuildOntologyLayer1` (`api/prospects.js:145-146`) deletes `ontology_entities WHERE layer = 1`. Company entities are layer 1, the Layer 2 import reuses them **without** bumping their layer (`:2796-2815`), and `ontology_relationships` FKs are `ON DELETE CASCADE` (`scripts/create-ontology-tables.sql:51-52`). So every full rebuild cascades away **all Layer 2 relationships whose subject is a company** — i.e. essentially all manually-extracted Layer 2 data. Full rebuilds auto-run after **every bulk import** (`:3347`). Directly contradicts the documented guarantee "Layer 1 rebuild does NOT delete Layer 2 data."
**Fix:** delete only layer-1 *relationships* in the full rebuild (mirroring the per-prospect rebuild's discipline), or exclude entities referenced by layer-2 relationships.

### A1.4 — No rate limiting on login; 6-digit PIN is brute-forceable
`api/auth.js:54-91`: unlimited attempts, no lockout, no backoff, against a 1,000,000-key space, on infinitely-parallel serverless. Sessions never expire, so one cracked PIN is permanent. Compounding side-channels: unknown email returns *before* the ~80ms bcrypt compare (timing reveals which emails exist), and deactivated accounts get a distinct error message (`:71`). PINs are also generated with `Math.random()` (`:47`, `:367`) rather than `crypto.randomInt`.
**Fix:** failed-attempt counter (per email) with lockout/backoff; dummy bcrypt compare on the unknown-email path; generic error for deactivated accounts; `crypto.randomInt`.

### A1.5 — Real contact spreadsheet is publicly downloadable
`public/cwp plastics contacts 3-16-26.xlsx` (890 KB, git-tracked) deploys as a Vercel static asset — downloadable by anyone at `/cwp plastics contacts 3-16-26.xlsx`, no login. Zero code references it; it serves no function.
**Fix:** delete from `public/` (note it remains in git history; rewriting history is optional but worth knowing).

### A1.6 — `api/bulk-upload-reports.js`: unauthenticated destructive leftover
Its own header (line 5) says "DELETE THIS FILE after successful upload." Still deployed: GET serves a public HTML page with a "Run Upload" button; unauthenticated POST archives every current state report (`is_current=false`) and re-inserts 44+ reports fetched from a hardcoded, possibly-dead git branch URL. Also the undocumented **11th of 12** Vercel function slots.
**Fix:** delete the file.

# C. Everything Else Worth Knowing

**C1. Zero automated tests.** No test files exist (`*.test.*`/`*.spec.*` count: 0), no test runner configured. Every regression so far has been caught by humans clicking. Given the documented SYNC-pair pattern, even a tiny Node test suite that asserts the pairs match (priority score fixtures, corridor maps, urgency thresholds, category rules) would catch the exact class of drift this audit found — several SYNC pairs *have already drifted* (corridors, EXCEL_TO_DB, and the badge pair degenerated to one side).

**C2. The database schema is not reproducible.** No committed DDL exists for ~9 live tables (`opportunities`, `stage_transitions`, `deadlines`, `meeting_minutes`, `prospect_attachments`, `prospect_activity_log`, `state_research_reports`, `contacts`, `stakeholder_alerts`) or for ~12 later-added columns (`priority_score`, `needs_review`, `digest_*`, etc.) — they were applied ad-hoc in the Neon console. A fresh environment built from `scripts/` cannot run the app. Also `scripts/create-prospect-table.sql` lacks the `country` column that `seed-prospects.js` inserts → bootstrap-from-scratch fails. Worth one session: snapshot the live schema (`pg_dump --schema-only`) into `scripts/` as the canonical baseline.

**C3. Dependency vulnerabilities (`npm audit`: 4 high, 1 moderate).** `vite` dev-server issues — fix with `npm audit fix` (build output unaffected, but cheap to clear). `xlsx` (SheetJS) prototype pollution + ReDoS — **no fix available on the npm registry**; SheetJS now ships fixed builds only via their own CDN registry (`https://cdn.sheetjs.com/xlsx-latest/xlsx-latest.tgz`). Risk is moderate (file parsing is client-side, on files your own team uploads), but switching the install source is a one-line `package.json` change.

**C4. Bundle: single 1.35 MB JS chunk (415 KB gz).** `xlsx`, `d3`, `chart.js`, and `react-markdown` all load up-front for every user on every visit. Dynamic `import()` for BulkImportModal (xlsx) and the graph views (d3) would cut initial load roughly in half. Cosmetic at 4 users, but free wins.

**C5. Function slots: 11 of 12 used.** Deleting `bulk-upload-reports.js` → 10. `analytics.js`, `health.js`, and (pending confirmation) `meeting-minutes.js` are idle slots if you ever need room.

**C6. Concurrency model is last-write-wins with no conflict detection.** Acceptable for 4 users, but the failure mode is silent data loss (two people editing the same prospect, background refetch clobbering an in-progress `EditableField` draft — A3). A cheap `updated_at` precondition on PATCH would convert silent loss into a visible "this row changed under you" error.

**C7. CLAUDE.md has drifted in 15+ places.** This matters more here than in a normal repo: **Claude does the coding, so CLAUDE.md is effectively the spec** — wrong docs become wrong future code. Corrections needed: function count (11, not 10, with `bulk-upload-reports` unlisted); phantom paths (`src/lib/*`, `src/hooks/`, `src/pages/`, `src/components/ui/`, `src/components/opportunities/`, `ForceGraphTestPage.jsx`); "same EXCEL_TO_DB mapping" (diverged); corridor table omits DE/MD/WV; corridors.js "imported by StateTooltip" (it isn't); the stale Follow-Up Tracking section (Action Items preset and Due column no longer exist — contradicts the newer Tasks section); undocumented auth actions (`change-pin`, `team-members`, `setup`) and ChangePinModal; undocumented prospects actions (`seed`, `apply-parent-type-classifications`, `backfill-parent-types-dryrun`); parseLocalDate has 3 copies, not 2 (`taskUtils.js:8`); "analytics accepts same filter params" (it lacks `prospect_status`); export-json line-ref (`:2527`, not `:2356`); "the ONE localStorage usage" (there are three UI-pref keys); Ownership Urgency table omits the `corporate/strategic → 2` branch and documents a PE contract the dropdown value can't satisfy; ontology relationship-type count (28 seeded, docs/comment say 23); search-fields list missing `former_names`/`financial_sponsor`/`country`.

# D. Things That Will Come Back to Haunt Us

1. **The open API + public spreadsheet, the moment this URL circulates.** Vercel URLs get scanned, shared in emails, pasted into decks. Today the blast radius is "all alliance research + contacts readable and writable by anyone with the URL." It's also the kind of finding a university IT/security review flags instantly.
2. **Layer 2 ontology loss (A1.3).** The first time someone invests real hours in Layer 2 extraction and then runs a bulk import, that work is gone — and nobody will notice until they query for it. Silent, delayed, unrecoverable.
3. **No schema migrations (C2).** The day Neon has an incident, or you want a staging environment, or you onboard another developer/AI session against a fresh DB — there is no path to recreate the database. This is the single biggest "bus factor" item.
4. **SYNC-by-comment without tests (C1).** Three pairs have already drifted. Every future feature that touches corridors, scoring, urgency, or import mappings re-rolls these dice.
5. **The 10s timeout on bulk import (A2.17).** Works at 179 companies, fails somewhere north of that — and when it fails it half-wipes the ontology. It will break precisely when the prospect database grows, i.e., when the tool is succeeding.
6. **Vercel function ceiling (C5).** At 11/12, the *next* feature that needs an endpoint forces a refactor under deadline pressure. Deleting the leftover gets you breathing room now.
7. **`xlsx` dependency (C3).** Unfixable via npm; every future `npm audit` stays red until the install source changes, training everyone to ignore audits.
8. **Sole-admin lockout (A3).** One mis-click in the admin panel by Kyle = nobody can administer the app, recoverable only via direct DB surgery.
9. **CLAUDE.md drift (C7).** In an AI-built codebase, doc rot compounds: each inaccurate claim seeds future sessions with false premises (this audit found code written to match docs that no longer matched reality).
10. **Last-write-wins (C6).** Fine at 4 users — until the week two people work the same Group 1 list and quietly overwrite each other's notes before an outreach push.
